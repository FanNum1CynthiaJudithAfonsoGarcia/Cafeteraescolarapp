1. Introducción

1.1 Presentación del proyecto

Cafetería Escolar App es una aplicación web fullstack diseñada para digitalizar y optimizar el proceso de pedidos de una cafetería de centro escolar. Permite a los alumnos realizar sus pedidos de forma anticipada desde cualquier dispositivo con navegador, mientras que el personal de administración gestiona el catálogo de productos, los usuarios registrados y el historial de pedidos desde un panel dedicado.

La aplicación cubre la totalidad del flujo de compra: registro e identificación del usuario, selección de productos por categoría con opciones de personalización, carrito de compra, pago en línea con tarjeta y confirmación con número de pedido. Opcionalmente, imprime un ticket físico en una impresora térmica conectada a la red local.

1.2 Contexto y necesidad de la solución propuesta

En muchos centros escolares, el proceso de compra en la cafetería genera largas colas a la hora del recreo, especialmente en los primeros minutos. Los alumnos pierden tiempo de descanso esperando, el personal de la cafetería se ve desbordado y los pedidos se pueden confundir o perder.

La digitalización del proceso permite a los alumnos realizar su pedido antes del recreo, reduciendo la espera a un simple momento de recogida en el mostrador. Además, el control administrativo de usuarios e historial de pedidos facilita la gestión económica del negocio.

1.3 Funcionalidades principales del sistema

El sistema define tres tipos de usuario con distintos privilegios:

RolDescripción
CLIENTEAlumno del centro. Puede consultar el catálogo, hacer pedidos y ver su historial.
TUTORDocente. Tiene las mismas capacidades que el cliente. Se registra con un código secreto.
ADMINPersonal de la cafetería. Gestiona todo el sistema desde el panel de administración.

2. Objetivos del Proyecto

2.1 Objetivo general

Desarrollar una aplicación web que permita a los alumnos de un centro escolar realizar pedidos anticipados en la cafetería, reduciendo las colas durante el recreo y mejorando la eficiencia del servicio mediante la digitalización completa del proceso de compra.

2.2 Objetivos específicos

  • Autenticación segura por roles: Implementar un sistema de registro y login con tres roles diferenciados (CLIENTE, TUTOR, ADMIN) y contraseñas cifradas con bcrypt.
  • Catálogo dinámico: Mostrar los productos agrupados por categoría, recuperados desde la base de datos en tiempo real.
  • Carrito personalizable: Permitir a los usuarios añadir productos con opciones de personalización (tipos de pan, extras, ingredientes a eliminar).
  • Pago en línea: Integrar la pasarela de pago Stripe para aceptar pagos con tarjeta de forma segura.
  • Panel de administración: Proporcionar al administrador herramientas para gestionar pedidos, usuarios y el catálogo de productos sin necesidad de acceder directamente a la base de datos.
  • Impresión de tickets: Generar e imprimir tickets en formato ESC/POS en una impresora térmica de red al confirmar cada pedido.
  • Diseño responsive: Garantizar una experiencia de usuario óptima tanto en dispositivos móviles como en escritorio.

3. Contexto y Justificación

3.1 Problema detectado o necesidad a cubrir

El proceso tradicional de compra en la cafetería escolar presenta los siguientes problemas:

  • Colas excesivas: La concentración de todos los alumnos en el mismo momento del recreo genera tiempos de espera de hasta 15-20 minutos.
  • Errores en pedidos: La comunicación verbal de pedidos complejos (bocadillos personalizados, restricciones alimentarias) provoca errores frecuentes.
  • Falta de trazabilidad: No existe registro digital de las ventas, dificultando el control económico y el análisis de la demanda.
  • Gestión de alergenos manual: Los alumnos con alergias alimentarias deben recordar comunicarlas en cada compra, con el riesgo que ello conlleva.

3.2 Justificación tecnológica del desarrollo

Se ha optado por una arquitectura web SPA (Single Page Application) por los siguientes motivos:

  • Accesibilidad universal: Funciona en cualquier dispositivo con navegador sin necesidad de instalación.
  • Prisma ORM: Abstrae el acceso a la base de datos, facilitando el mantenimiento y la posible migración de gestor (MySQL → PostgreSQL, SQLite…) con cambios mínimos.
  • Stripe: Es la solución de pagos en línea más utilizada a nivel empresarial, con amplia documentación y cumplimiento PCI-DSS.
  • TiDB Cloud (MySQL): Base de datos relacional en la nube con alta disponibilidad, sin necesidad de gestionar infraestructura de servidor de BD.
  • TypeScript en ambas capas: Añade tipado estático, reduciendo errores en tiempo de desarrollo y mejorando la mantenibilidad.

3.3 Impacto esperado y valor aportado

  • Reducción del tiempo de espera en el mostrador a menos de 2 minutos por alumno.
  • Eliminación de errores en pedidos personalizados al registrarlos digitalmente.
  • Registro automático de todas las transacciones para control financiero.
  • Gestión centralizada de alergenos por usuario en la base de datos.

5. Análisis de Requisitos

5.1 Requisitos funcionales

RF-01 — Gestión de usuarios

  • El sistema debe permitir el registro de nuevos usuarios con los campos: nombre completo, email (único), contraseña (mínimo hash bcrypt), rol, horario y alergenos.
  • El sistema debe validar el código secreto para los roles TUTOR y ADMIN.
  • El sistema debe autenticar usuarios mediante email y contraseña.
  • La sesión debe persistir entre recargas de página mediante localStorage.

RF-02 — Catálogo de productos

  • El sistema debe mostrar las categorías de productos disponibles en la página de inicio.
  • El sistema debe mostrar los productos pertenecientes a una categoría seleccionada.
  • Cada producto debe mostrar: nombre, precio, imagen, descripción y alergenos.
  • El sistema debe permitir personalizar ciertos productos (bocadillos: tipo de pan, extras, sin ingredientes).

RF-03 — Carrito de compra

  • El sistema debe permitir añadir productos al carrito con su cantidad y opciones de personalización.
  • El sistema debe calcular el precio total incluyendo extras.
  • El usuario debe poder modificar cantidades y eliminar productos del carrito.

RF-04 — Proceso de pago

  • El sistema debe crear un PaymentIntent en Stripe con el importe calculado en servidor.
  • El sistema debe mostrar un formulario de tarjeta seguro (Stripe Elements).
  • Tras el pago exitoso, el sistema debe crear el pedido en la base de datos.
  • El sistema debe redirigir a una página de confirmación con el número de pedido.

RF-05 — Historial

  • El sistema debe mostrar al usuario todos sus pedidos anteriores con fecha, artículos y total.

RF-06 — Administración

  • El ADMIN debe poder ver todos los pedidos con búsqueda por usuario/email.
  • El ADMIN debe poder ver, buscar y eliminar usuarios (no a sí mismo).
  • El ADMIN debe poder ver, crear y eliminar productos del catálogo.

RF-07 — Impresión

  • El sistema debe intentar imprimir un ticket ESC/POS en la IP configurada al confirmar el pedido.
  • Si la impresora no está disponible, el sistema debe continuar sin error para el usuario.

6. Diseño de la Solución

6.1 Arquitectura del sistema

La aplicación sigue una arquitectura cliente-servidor desacoplada (SPA + API REST):

Modelo de comunicación:

  • El frontend (React SPA) se comunica con el backend (Express) mediante peticiones HTTP REST.
  • El backend es el único que accede a la base de datos (Prisma → MySQL/TiDB Cloud).
  • El pago se gestiona entre el frontend y Stripe directamente, usando el client_secret generado por el backend.
  • La impresora térmica se conecta directamente al backend mediante TCP.

Tecnologías base en cada capa:

CapaTecnología

ComunicaciónFetch API (cliente) / Express Router (servidor)

Base de datosMySQL 8 en TiDB Cloud (eu-central-1)
Autenticaciónbcryptjs + localStorage
PagosStripe SDK (cliente + servidor)
Impresiónnode-thermal-printer (ESC/POS por TCP)

Herramientas complementarias:

  • dotenv — carga de variables de entorno en el servidor
  • cors — habilitación CORS entre frontend (:5173) y backend (:3000)
  • nodemon + ts-node — ejecución y hot-reload en desarrollo
  • Vite — bundler del frontend con HMR

7.3 Integración de módulos / componentes

Integración cliente-servidor:

El servicio src/services/api.ts centraliza todas las llamadas HTTP al backend. Cada función mapea a un endpoint REST:

`

GET /api/productos → getProductos()

GET /api/productos/categorias → getCategorias()

POST /api/productos → createProductoAdmin()

DELETE /api/productos/:id → deleteProductoAdmin()

POST /api/usuarios/login → loginUsuario()

POST /api/usuarios → registerUsuario()

GET /api/usuarios/admin/todos → getAllUsuariosAdmin()

DELETE /api/usuarios/:id → deleteUsuarioAdmin()

POST /api/pedidos → createPedido()

GET /api/pedidos/usuario/:id → getPedidosUsuario()

GET /api/pedidos/admin → getAllPedidosAdmin()

POST /api/pagos/create-payment-intent → (Stripe)

`

Integración de servicios externos — Stripe:

  • El frontend llama a POST /api/pagos/create-payment-intent con los artículos del carrito.
  • El backend recalcula el total consultando los precios reales en la base de datos (seguridad).
  • El backend crea un PaymentIntent en Stripe y devuelve el clientSecret.
  • El frontend usa Stripe Elements con ese clientSecret para mostrar el formulario de tarjeta y confirmar el pago directamente con Stripe.

Integración de la impresora térmica:

  • La IP y el puerto de la impresora se configuran en el .env del servidor (PRINTER_IP, PRINTER_PORT).
  • Al confirmarse un pedido, el controlador llama a imprimirTicket() del módulo src/lib/printer.ts.
  • El módulo intenta conectar por TCP; si no hay conexión, registra un console.warn y continúa sin bloquear al usuario.

Validaciones y seguridad:

ValidaciónDónde
Campos obligatorios (nombre, email, password, horario)usuario.controller.ts
Código secreto para roles TUTOR y ADMINusuario.controller.ts

Importe mínimo Stripe (≥ 0.50€)paymentController.ts

8.2 Pruebas no funcionales

Rendimiento:

  • El singleton de Prisma evita abrir múltiples conexiones durante el desarrollo con nodemon (hot-reload).
  • Las peticiones a la API responden en < 500ms en condiciones normales de red.
  • El bundle de producción generado por Vite es optimizado con code-splitting automático.

Seguridad:

  • Verificado que el campo password nunca se incluye en las respuestas de /api/usuarios.
  • Los precios se recalculan en servidor en createPedido y crearIntentoDePago, evitando que un cliente manipule los importes.
  • Las claves API de Stripe y la URL de base de datos están en .env y no se incluyen en el repositorio Git (.gitignore).
  • Las contraseñas se almacenan cifradas con bcrypt (salt rounds: 10), verificado inspeccionando la base de datos directamente.

Usabilidad:

  • Todos los botones de acción muestran spinner de carga durante la petición HTTP.
  • Los errores de red y de validación se muestran con mensajes descriptivos en la interfaz.
  • La aplicación es completamente usable en pantallas de 375px (móvil) y 1920px (escritorio).
  • El panel de administración adapta su navegación: sidebar lateral en escritorio, barra inferior en móvil.

9. Conclusiones y Líneas de Mejora

9.1 Resultados obtenidos

Se ha logrado implementar una aplicación web fullstack funcional que cubre íntegramente el flujo de pedido en una cafetería escolar, desde el registro del usuario hasta la confirmación del pago y la generación del ticket. La integración de todos los módulos (React + Express + Prisma ORM + TiDB Cloud + Stripe + impresora térmica) funciona de forma coordinada y el sistema es estable en el entorno de desarrollo local.

Durante el desarrollo del proyecto se han aplicado y consolidado conocimientos de todos los módulos del ciclo: programación orientada a objetos (TypeScript), acceso a datos (Prisma + MySQL), desarrollo web en entorno cliente (React + Vite), desarrollo web en entorno servidor (Node.js + Express), y servicios de red (API REST, TCP, HTTPS).

9.2 Limitaciones detectadas

LimitaciónDescripción
Autenticación básicaLa sesión se gestiona con localStorage sin tokens JWT, lo que no es apto para producción sin mejoras
Singleton Prisma en paymentpaymentController.ts instancia su propio PrismaClient en lugar de usar el singleton compartido, generando conexiones extra
Sin validación de stockNo existe un campo de cantidad en stock; los productos simplemente tienen un flag disponible
Sin gestión de estado del pedidoNo hay estados de pedido (pendiente, preparando, listo, entregado)
Sin notificaciones en tiempo realNo hay WebSockets ni Server-Sent Events; el admin debe recargar para ver nuevos pedidos
Imágenes estáticasLas imágenes de productos son rutas locales (/CafeSolo.png), no existe gestión de ficheros en servidor

9.3 Propuestas de mejora y trabajo futuro

  • Autenticación con JWT: Implementar tokens de acceso firmados con expiración y refresh tokens.
  • WebSockets para tiempo real: Notificar al administrador de nuevos pedidos al instante.
  • Estado de pedido con seguimiento: Añadir un ciclo de vida (PENDIENTE → PREPARANDO → LISTO → ENTREGADO) visible tanto para el cliente como para el administrador.
  • Gestión de imágenes: Integrar un servicio de almacenamiento (Cloudinary, AWS S3) para subir imágenes desde el panel de administración.
  • Tests automatizados: Implementar tests unitarios con Vitest (frontend) y Jest (backend), y tests de integración con Supertest.
  • Control de stock: Añadir un campo stock a los productos y decrementarlo automáticamente con cada pedido.
  • Despliegue en producción: Contenerizar con Docker y desplegar el backend en un servicio como Railway o Render, y el frontend en Vercel o Netlify.
  • Aplicación móvil nativa: Desarrollar una app React Native aprovechando la API REST ya existente.

sequenceDiagram

actor U as Usuario

participant FE as Frontend React

participant ST as Stripe API

participant DB as TiDB Cloud (MySQL)

participant PR as Impresora

BE->>DB: SELECT precio FROM productos WHERE id IN (...)

DB-->>BE: precios reales

BE->>ST: paymentIntents.create({amount, currency:'eur'})

ST-->>BE: {clientSecret}

BE-->>FE: {clientSecret}

FE->>U: Muestra formulario Stripe Elements

U->>FE: Introduce datos de tarjeta y confirma

FE->>ST: stripe.confirmPayment({clientSecret})

ST-->>FE: {status: 'succeeded'}

FE->>BE: POST /api/pedidos {usuarioId, items}

BE->>DB: INSERT INTO pedidos + detalle_pedidos (transacción)

DB-->>BE: pedido creado {id}

BE->>PR: TCP print ticket PED-{id}

BE-->>FE: 201 {pedido}

FE->>U: Redirige a /pago-exitoso
